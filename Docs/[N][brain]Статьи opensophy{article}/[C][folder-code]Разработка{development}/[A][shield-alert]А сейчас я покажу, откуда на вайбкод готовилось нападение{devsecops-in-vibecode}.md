# Часть 1 — контекст, проблема, эксперимент

## Предисловие

В июле 2025 года инженер из команды Claude Code в Anthropic — **Brian Cherny** — написал в X следующее:

> *«Pretty much 100% of our code is written by Claude Code + Opus 4.5. For me personally it has been 100% for two+ months now, I don't even make small edits by hand. I shipped 22 PRs yesterday and 27 the day before, each one 100% written by Claude.»*
>
> — [Brian Cherny, @bcherny](https://x.com/bcherny/status/2015979257038831967)

Дальше он добавил: большинство индустрии придёт к похожей статистике в ближайшие месяцы. И что они уже используют `claude -p` на каждом PR для code review в свежем контексте — это помогает ловить и исправлять многие проблемы.

Инженер из команды, которая буквально делает Claude, говорит, что 100% кода пишет Claude. Не «помогает», не «автодополняет» — пишет. 22 PR в день. И это уже сейчас, не в будущем.

И здесь у любого инженера, думающего о безопасности, должен загореться один большой вопрос: хорошо, ЛЛМ пишет код — **а кто проверяет его безопасность?**

## Откуда растут ноги у проблемы

В марте 2026 года компания DryRun Security опубликовала **«The Agentic Coding Security Report»** — небольшой, но очень показательный документ. Они взяли трёх кодинг-агентов — **Claude** (Sonnet 4.6), **Codex** (GPT 5.2) и **Gemini** (2.5 Pro) — и дали каждому одно и то же задание: написать два приложения с нуля, добавляя фичи через pull request'ы, как это делает реальная команда разработчиков.

После каждого PR прогонялся сканер безопасности. Результаты оказались, мягко говоря, интересными.

**Что нашли:**

- **87% pull request'ов** содержали хотя бы одну уязвимость
- **143 проблемы безопасности** суммарно из 38 сканов
- Одни и те же классы уязвимостей воспроизводились у всех трёх агентов независимо друг от друга

Топ повторяющихся проблем:

| Класс уязвимости | Уровень | Кто допустил |
|---|---|---|
| Broken Access Control | Высокий | Все три агента |
| Hardcoded JWT secrets | Высокий | Все три агента |
| WebSocket Auth Bypass | Высокий | Все три агента |
| IDOR | Высокий | Все три агента |
| User Enumeration | Средний | Все три агента |
| XSS | Средний | Gemini + все в отдельных случаях |
| Race Conditions (TOCTOU) | Средний | Преимущественно Gemini |

Отдельно бросается в глаза одна деталь: агенты **правильно писали middleware для аутентификации** в REST API — и тут же забывали подключить его к WebSocket-хендлеру. Middleware существовал, был написан корректно, но попросту не работал. Это не баг в логике — это ошибка архитектурного контекста, которую традиционные SAST-инструменты на основе паттернов не поймают.

Лучшие результаты показал Codex — но и у него остались незакрытые JWT-проблемы и отсутствие ревокации токенов.

**Вывод из отчёта простой:** ЛЛМ хоть и пишут неплохо код но все же стоит проверять на безопасность и качество.

## А теперь эксперимент

В качестве практики и эксперимента был создан **[Opensophy Hub](https://hub.opensophy.com)** — open-source статическая платформа для документации с богатым набором Markdown blocks, dev-панель, интерактивными таблицами, UI-компонентами и live-редактором. Задача была в том, чтобы [решать реальные задачи с помощью ИИ](https://habr.com/ru/news/1013212/) и посмотреть, как это работает на практике.

Claude писал код — компоненты, логику, скрипты, парсер Markdown, систему навигации, dev-панель. Я в роли «архитектора» занимался архитектурными решениями: структура проекта, выбор паттернов, ревью результатов и контроль общего направления.

И именно на его примере я покажу, как можно выстроить DevSecOps-пайплайн для вайбкодерского проекта — с реальными инструментами, реальными проблемами и реальными исправлениями.

## Что такое Quality Gate и зачем он нужен

Прежде чем переходить к практике, разберёмся с термином **Quality Gate** (ворота качества).

Quality Gate — это набор условий, при невыполнении которых сборка или деплой автоматически блокируются. Грубо говоря, это контракт: «если код не соответствует этим критериям — он не едет в продакшн». Типичные условия:

- покрытие тестами не ниже N%
- количество новых уязвимостей = 0
- нет критических code smell'ов
- оценка надёжности не ниже B

В нашем случае SonarCloud будет проверять код на каждый push и выставлять оценку. Если Quality Gate не пройден — GitHub Actions сигнализирует об ошибке, деплой не происходит. Среда просто не обновляется, пока проблема не устранена.

## Двойная ИИ-интеграция: идея эксперимента

К самой интересной части. Мы говорим о проекте, написанном преимущественно ИИ. Классический подход «разработчик пишет → SAST проверяет» здесь трансформируется:

```
Claude Sonnet 4.5
   ↓ пишет код Hub
SonarCloud
   ↓ анализирует статически, находит проблемы
Claude Sonnet 4.6
   ↓ читает отчёты, исправляет найденные проблемы
```

Модель генерирует компонент, SonarCloud находит в нём, например, потенциальный ReDoS в регулярном выражении или небезопасный паттерн обработки данных, я передаю отчёт в Claude 4.6 с промптом «исправь эти проблемы, сохрани логику» — и получаю исправленный код.

Почему две разные версии, а не одна? Во-первых, у 4.6 лучше рассуждения при анализе существующего кода. Во-вторых, это принципиально соответствует идее **независимой проверки**: один агент создаёт, другой ревьюит. В идеале сюда добавить ещё третий — например, Codex — чтобы получить несколько точек зрения и минимизировать проблему «одна модель = одни и те же ошибки».

Важно понимать: ИИ-агент в роли ревьюера не заменяет инструменты. SonarCloud всё равно нужен — он даёт **структурированный отчёт** с конкретными строками, severity и типом проблемы. Claude работает не с абстрактным «посмотри на код», а с конкретным списком проблем. Это принципиальная разница.

# Часть 2 — подключение SonarCloud и настройка Quality Gate

## Способы подключить SonarCloud

После регистрации проекта в SonarCloud и первичного сканирования переходим в **Administration → Analysis method** — там будут перечислены все доступные варианты интеграции:

- **Automatic analysis** — облачный автоматический анализ без CI
- **GitHub Actions** — сканирование запускается как шаг в вашем workflow
- **CircleCI и другие CI-инструменты** — аналогично, но для других платформ
- **Manual** — для локального тестирования, запускается вручную

> Для каждого из этих вариантов SonarCloud предоставляет готовые инструкции прямо в интерфейсе — **Administration → Analysis method**. После выбора нужного метода там будет пошаговый гайд с конкретными командами и конфигурационными файлами под ваш стек.

![Текст](/assets/vibecode6.png "Вот тут настройки выбора анализа.")

### Автоматический анализ — наш выбор

На первый взгляд может показаться, что GitHub Actions — очевидный выбор: больше контроля, явная интеграция в пайплайн. Но для большинства проектов, особенно на старте, **автоматический анализ предпочтительнее** и вот почему: 

Automatic analysis подключается напрямую к репозиторию через GitHub App: никаких дополнительных workflow-файлов, секретов и ручной настройки. SonarCloud сам запускает анализ при каждом пуше и подтягивает результаты прямо в pull request — статус, количество проблем, ссылка на дашборд. Работает из коробки.

Именно его и будем использовать как основной способ. Он покрывает подавляющее большинство сценариев: статический анализ кода, выявление уязвимостей, code smell'ы, дублирование — всё это доступно без единой строчки в YAML.

### Когда нужен GitHub Actions

GitHub Actions имеет смысл, если вам нужно **жёстко заблокировать деплой** при непрохождении Quality Gate. Automatic analysis работает асинхронно: SonarCloud анализирует код и сообщает результат, но не может остановить уже запущенный деплой. Если между пушем и завершением анализа проходит хотя бы минута — деплой уйдёт вперёд.

С GitHub Actions можно выстроить линейную цепочку:

```
push → sonarqube job → build → deploy
```

Где `build` физически не запустится, пока `sonarqube` не завершится с успехом. Ключевой механизм здесь — флаг `sonar.qualitygate.wait=true`. Без него сканер отправляет данные и сразу завершается с кодом `0`, не дожидаясь результата Quality Gate. С флагом — шаг висит и ждёт: получил `FAILED`, упал с exit code `3`, и вся цепочка останавливается.

Пример итогового `deploy.yml` с интеграцией SonarCloud для Hub:

```yaml
name: Deploy to GitHub Pages

on:
  push:
    branches: [main, master]
  pull_request:
    types: [opened, synchronize, reopened]
  workflow_dispatch:

concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  sonarqube:
    name: SonarQube
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # Нужен полный history для корректного diff-анализа
      - name: SonarQube Scan
        uses: SonarSource/sonarqube-scan-action@v6
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
        with:
          args: >
            -Dsonar.qualitygate.wait=true

  build:
    runs-on: ubuntu-latest
    needs: sonarqube  # Не запустится, пока sonarqube не пройдёт
    permissions:
      contents: read
      pages: write
    steps:
      - name: Checkout
        uses: actions/checkout@v4
      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: 'npm'
      - name: Install dependencies
        run: npm ci
      - name: Generate docs
        run: npm run generate:docs
      - name: Generate sitemap
        run: npm run generate:sitemap
      - name: Build
        run: npm run build
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./dist

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    permissions:
      pages: write
      id-token: write
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

Цепочка линейная: `sonarqube → build → deploy`. Если sonarqube упал — всё стоит.

Если Quality Gate не прошёл, в логах GitHub Actions появится примерно следующее:

```
ERROR QUALITY GATE STATUS: FAILED
Error: Action failed: The process '...sonar-scanner' failed with exit code 3
```

И ссылка на дашборд SonarCloud с конкретными проблемами — файл, строка, severity.

![Текст](/assets/vibecode1.png "Демонстрация работы Quality Gate(не прошел качество кода)")

### sonar-project.properties

Независимо от выбранного метода интеграции, в корне репозитория нужен файл `sonar-project.properties`. При выборе метода в SonarCloud его базовое содержимое будет сгенерировано автоматически — там уже будут `projectKey` и `organization`:

```properties
sonar.projectKey=opensophy-projects_hub
sonar.organization=opensophy-projects

# Исключаем зависимости и артефакты сборки из анализа
sonar.exclusions=**/node_modules/**,**/dist/**,**/public/**,**/scripts/**,**/.astro/**
```

Файл коммитим в репозиторий — сканер подхватит его автоматически.

## Настройка Quality Gate в SonarCloud

По умолчанию SonarCloud применяет **Sonar way** — стандартный Quality Gate. Для большинства проектов он вполне адекватен. Посмотреть и скастомизировать его можно в **Policies → Quality Gates**.

Стандартные условия Sonar way для нового кода:

| Метрика | Порог |
|---|---|
| Security Rating | A |
| Reliability Rating | A |
| Maintainability Rating | A |
| Coverage | ≥ 80% |
| Duplicated Lines | ≤ 3% |

Для проекта, который долго не проверялся, первый запуск может показать от 50 до 200+ проблем. Условие по Coverage при этом сразу убьёт Quality Gate — его можно либо убрать из условий, либо выставить в 0%. Остальное имеет смысл оставить: Security и Reliability Rating на уровне A означают «новый код не добавляет уязвимостей и явных багов» — это разумная планка для любого проекта. Затем можно уже подключать условие по Coverage но это по вашему усмотрению как вы хотите.

В дальнейшем можно расширить набор инструментов: добавить SCA через Snyk или GitHub Dependabot для проверки зависимостей, а для DAST-анализа попробовать HostedScan — там есть несколько бесплатных сканов OWASP ZAP с отчётом в удобном формате, который можно сразу передать в ИИ для анализа.

# Часть 3 — реальная проверка кода Hub, отчёты, исправления через Claude 4.6

Итак. Допустим, ваш проект уже готов к релизу, SonarCloud подключён, и вы только что посмотрели на результаты первого полноценного анализа:

![Текст](/assets/vibecode7.png "234 проблем в коде!!")

Жёстко, да? Это примерный результат для проекта с историей — когда впервые запускаешь анализ после долгой разработки. Но если смотреть на факты: большинство из этих 234 проблем — code smell'ы. Явные баги ИИ пишет сравнительно редко, но качество кода страдает, особенно когда не до конца понимаешь, что именно модель сгенерировала в конкретном месте.

Переходим к демонстрации устранения проблем.

## Убираем проблемы в коде благодаря Claude Sonnet 4.6

Работать с исправлениями можно через любой интерфейс — IDE, API, CLI. Лично мне удобнее **веб-версия Claude**: можно одновременно видеть ход рассуждений модели, прикладывать скриншоты и сразу получать готовый файл. К тому же базовый план бесплатный.

### Шаг 1. Промпт

```markdown
SonarCloud сообщает о проблемах в коде. Проверь и полностью исправь файл. Твоя задача — сделать код максимально простым, читаемым и оптимизированным. При этом учитывай, что необходимо проверять предупреждения на ложные срабатывания (false positive).
Примечание: все комментарии должны быть на русском языке. Не добавляй лишние комментарии — только по функционалу. Никаких комментариев об исправлениях.
```

Почему такие требования к комментариям: если не указать явно, модель склонна добавлять пометки вида `// FIX: исправлен nested ternary` или `// SONAR: S3358`. Это мусор в коде — он не несёт функциональной ценности и захламляет историю. К примеру модель исправила код но не до конца или сделала другие ошибки при изменении файла, из-за чего в нашем случае код будет еще больше из-за комментариев.

### Шаг 2. Код

Копируем содержимое файла с проблемами и вставляем в чат. Для больших файлов Claude автоматически переключается в режим `pasted` — это нормально, контекст сохраняется.

### Шаг 3. Скриншоты проблем

Переходим в SonarCloud → **Issues** или в **Security Hotspots** (если говорить по приоритету то лучше решить сразу Security Hotspots), делаем скриншоты нужных проблем и прикладываем к сообщению:

![Текст](/assets/videcode10.png "Примерно получается так по шагам.)")

Важный момент: одна и та же проблема может встречаться в файле несколько раз. Например, `Prefer globalThis over window` у меня было 9 вхождений. Если все они видны на скриншоте — отлично. Если нет — возможно стоит явно указать в промпте: «исправь все вхождения этой проблемы по всему файлу».

По этой же схеме проходим через Security Hotspots и остальные категории issues — делаем до тех пор, пока список не опустеет.

![Текст](/assets/vibecode8.png "Спустя пару часов исправлений")

В результате получаем код, который прошёл статический анализ и соответствует заданному Quality Gate. Это не значит, что он идеален — но это значит, что он объективно лучше, чем до проверки.

# Часть 4 — добавляем еще инструменты и посмотрим на их минусы в процессе

Окей допустим все Части мы прошли(поставили SonarQube Cloud и обсудили ci/cd варианты а также принцип работы и исправления кода.) Давайте теперь продолжим усиливать качество проекта и как пример добавим еще один SAST инструмент это Semgrep, опытные профи могут сказать что SonarQube Cloud он больше про "качество кода" а нам нужно еще и "безопасность" проверить. И как раз Semgrep — это инструмент, который в первую очередь сфокусирован на безопасности и на этом примере мы попытаемся сейчас показать проблемы когда у нас несколько сканеров. (на заметку: Semgrep тоже бесплатный тариф имеет облачный)

## Регистрация в Semgrep 

Впринципе регистрация аналогична как и у SonarQube Cloud но только учитываем что все данные будут в Code а добавление и запуск повторных сканирований будет в Projects.

## Сюрприз!

Хочу обратить внимание выше был скрин что у нас в SonarQube Cloud мы все исправили а вот в Semgrep нет! и на текущий момент есть 25 проблем. Давайте попробуем их устранить сохраняя при этом функциональность.

Что это значит? Нам нужно заставить ИИ таким образом сделать задачу чтобы она не сделала другие ошибки которые будут в SonarQube Cloud, из-за чего может появиться вопрос "а почему вот так случиться?" Правильно и точное объяснить этот момент не могу поскольку все зависит от промпта, запоминание контекста LLMки что она делала и понимание кода. Под пониманием кода я подразумеваю что код она исправляет строго там где нужно исправить именно проблемы сохраняя при этом функциональность. Простой пример если не поняли: допустим вы дали задачу в Codex (https://chatgpt.com/codex) исправить уязвимость и баги в файле. Он успешно исправляет но задевает при этом фронтенд часть и темную тему которая должна быть черной он делает синий. Из-за чего приходиться больше тратить время и токены на исправление данной ситуации. ## Итоги

Мы прошли путь от «ИИ написал код и задеплоил» до «ИИ написал код, SAST проверил, другой ИИ исправил, Quality Gate пропустил». Но статический анализ иногда не ловит архитектурные проблемы вроде WebSocket Auth Bypass из отчёта DryRun. Но это работающий, воспроизводимый процесс, который:

- снижает количество очевидных уязвимостей и code smell'ов
- даёт структурированный отчёт вместо абстрактного «код плохой»
- встраивается в пайплайн и работает автоматически на каждый коммит

Схема `агент генерирует → инструмент анализирует → агент исправляет` — это не замена code review. Это гигиена для проекта, где большую часть кода пишет ИИ. И лучше иметь её, чем не иметь ничего.

## Что бы ещё для идеала?

Если у вас проблемы архитектурного плана то стоит наверное выдать полный к коду моделям по типу Codex, Claude чтобы они проверили в **жесткой** форме ваш проект и могли бы найти проблемы, но и здесь есть проблемы когда модель не до конца находит проблемы поэтому обязательно ручной code review и доп. инструменты проверки. Но и учитывайте что модель опять по такому принципу перейдет в этап разработки и соответственно нужна очередная проверка с помощью инструментов.

Добавлять больше инстументов проверки качества кода и смотреть что они покажут. К примеру codefactor + sonarcloud неплохая связка сканеров тк codefactor может еще найти другие проблемы в коде (но чаще встречал false positive)
